What is Vajra?

One sentence Vajra wraps your AI coding assistant. You type vajra claude instead of claude. Your agent works exactly the same — but Vajra watches what it does, keeps the expensive parts cheap, records everything it changed, and lets you set rules it can't ignore.

You pick the AI coding agent you like — Claude Code, Codex, Cursor, Aider, whoever. Vajra sits between you and that agent. It doesn't replace the agent, doesn't change what it can do, and doesn't slow it down. It just adds a layer of control and visibility that the agent itself can't give you.

Think of it like a flight recorder for your coding sessions. The agent flies the plane; Vajra records everything that happens and tells you exactly what it cost.

The on-ramp (free forever)

Compression

When the agent runs cargo test and gets 180 lines of output, Vajra shrinks it to 3 lines — saving tokens and money. Failures always come through in full so the agent can fix them.

The hook (trust builder)

Honest meter

After every session Vajra prints exactly what it cost — broken down by model, cache type, and subagent. No estimates. Numbers straight from the log files.

The moat (the company)

Audit ledger

A permanent, tamper-proof record of what your agents did — which agent wrote which line, when, in which session. Stored in your repo. No cloud. No expiry.

The big idea — and the strategic bet

The core insight Put the discipline in the environment, not in the prompt. Most "agent workflows" are rules pasted into the system prompt — re-read every session, cost tokens, and the agent can still ignore them. Vajra enforces rules around the agent, so they're free to apply and impossible to bypass.
What v1 ships (the trojan horse)

Free compression

Shortens noisy shell output before the agent sees it. Saves ~6–8% on your bill. Small but real, honest, and it installs the plumbing that the real product needs.

This is NOT the value prop. It's the reason to install Vajra.

What the company bets on

Governance + audit ledger

A tamper-proof record of everything every agent did, uniform across all agents, in an open standard format, stored permanently in your git repo — with policy enforcement on top.

No competitor does this cleanly. This is the moat.

Why compression is the right on-ramp even if it's not the product

  • It works on the first install with zero config — vajra claude and you're done.
  • The code that delivers compression (the hook that intercepts Claude Code's tool calls) is the same plumbing that v2 governance needs. So building compression builds the rail for free.
  • Showing users their real session cost, honestly labeled, builds the trust that makes them willing to install something that watches their agent.
  • Compression savings are small enough to be honest about — "~6–8%, here's why it's not more" is a credibility} signal, not a failure.

The economics — why compression saves less than you'd think

The pivotal finding (from brainstorm S3) Shortening shell output honestly saves about 6–8% of your total bill — not 50%, not 30%. This is real money but it's not the story. Here's why the number is small, and why that's fine.

How Claude Code billing actually works

When the agent runs a command and gets back 2,000 tokens of output, those tokens are charged full price once. After that, Claude Code caches them. Every subsequent turn that references that cached output costs only 10% of the original price. So if you compress 90% of that output, you save full price once — but the ongoing benefit is tiny because the cache was already doing the heavy lifting.

Token typePrice (relative)What compression saves
Output tokens (what the agent writes)25× the input priceNothing directly — compression doesn't reduce what the agent writes
Fresh input tokens (new content read this turn)1×Real — saves this one time
Cache writes (saving content for reuse, 5-min)1.25×Saved once when content first enters cache
Cache writes (1-hour tier)2×Saved once — bigger win
Cache reads (reusing cached content)0.1×Almost nothing — already dirt cheap
Web search / web fetchPer-request flat feeZero — not token-based
The real dollar lever The biggest savings aren't from shorter output — they're from preventing extra turns. If compression causes the agent to misunderstand a failure and loop through extra back-and-forth attempts, each of those output turns costs 25× more per token than any input compression saved. This is why we never compress small failure output: passing a failure through verbatim prevents the re-run loops that actually cost money.

The corrected cost formula (live-verified against Claude Code v2.1.177)

# Price for one line in a session log cost = ( input_tokens × model_input_price + output_tokens × model_output_price + cache_5min_tokens × model_input_price × 1.25 + cache_1hour_tokens × model_input_price × 2.00 + cache_read_tokens × model_input_price × 0.10 ) / 1,000,000 # Add per-request fees on top (not token-priced) cost += web_searches × price_per_search cost += web_fetches × price_per_fetch # Do this for every "assistant" line, by its specific model # Sessions mix models: main agent is Opus, titles/subagents use Haiku # Also parse every subagent log file — they live in a subfolder

Five traps that silently give you the wrong number

  • Don't add cache_creation_input_tokens — it's already the sum of the 5-min and 1-hour tiers. Adding it again double-counts.
  • Always open the subagent log files — Claude Code creates a separate log file for each subagent in a subfolder. Miss those files and you undercount the whole session.
  • Skip lines marked <synthetic> — these are placeholders, not real charges.
  • Don't add up the iterations array — it's a breakdown of the same numbers already counted at the top level, not additional charges.
  • Usage data is only on assistant lines — skip everything else.

The competition — and where we fit

The closest competitor: AxonFlow

AxonFlow is a Claude Code plugin that already ships ~80% of the governance vision we have: it intercepts tool calls, enforces 80+ built-in security policies, audits everything, and lets teams define custom rules. MIT licensed, free tier available. It's good. We have to be better in the right ways.

What AxonFlow does well — copy it
  • Zero-config setup — policies on by default, no friction
  • Inline explanations — when it blocks something, it tells you which rule and why
  • Override flow — you can create an approved exception with a reason, and that's audited too
  • Fast — adds less than 10ms per tool call
  • Good starter policy set — dangerous shell commands, SQL injection, exposed secrets
AxonFlow's weaknesses — this is our opening
  • Fails open on network loss — if their server is down, governance turns off. We run entirely on your machine.
  • Sends data to their cloud — tool inputs and outputs go to their servers for audit. We never leave your machine.
  • Free tier limited to 200 events/day — after that, governance stops or you pay. We're unlimited and free.
  • Audit logs expire after 3 days on free tier — an attacker can just wait. We store in your git repo, permanently.
  • Written in Bash — shell scripts in the security path are an attack surface. We ship a single compiled binary.

The open standard we adopt: agent-trace

Cursor (the IDE) published an open specification called agent-trace for recording which AI agent wrote which lines of code. It's a neutral format — any tool can read or write it. Instead of inventing our own format, we adopt it. We compete on being the best recorder of agent-trace data, not on having a different format.

Our exact opening We're the recorder + the ledger that nobody else has: write-time provenance at the line level, stored permanently in your git repo (not a SaaS database with a retention clock), in-process Rust engine (no network in the enforcement path), free and unlimited for solo developers.

The one-line moat

Sharpened positioning "The local-first, git-native, tamper-evident flight recorder for AI-written code — uniform across every agent, in the open agent-trace format — that also enforces policy, with no cloud and no retention cliff."

The success metric (not "% tokens saved")

We measure: median blended dollar cost with Vajra < without Vajra, on a realistic mix of tasks, with identical task outcomes (the agent succeeds at the same things), and zero forced VAJRA_RAW=1 re-runs.

We do NOT use "% input tokens reduced" — input tokens are cheap and cached. That number looks impressive and means almost nothing.

The day-1 failure we must avoid

The uninstall trap If Vajra ever silently hides a line the agent needed — a failing test name, an error message, an important log line — the agent will loop, waste turns, and produce wrong output. The user blames Vajra and uninstalls. This is the #1 product killer. Every compression rule is designed around this: when in doubt, pass through verbatim.

Design Decision 1 — How does compression reach Claude Code?

ADR-0001 · 2026-06-15 · Resolved the biggest open question from brainstorming

THE QUESTION
After an agent runs cargo test and gets 180 lines back — how do we shorten that output before the agent reads it?

Option A we rejected — The "shim" approach

Put a fake cargo binary early in the search path. When Claude Code runs cargo test, our fake binary catches it first, runs the real cargo, shortens the output, and passes the shorter version back.

The fatal flaw: our fake binary sits between two programs, not between the agent and the final result. So when the agent runs cargo test | tail -5 (a pipeline), our fake cargo feeds compressed output into tail — and tail now cuts the last 5 lines of compressed output, not the last 5 lines of the real output. Completely wrong. There's no clean fix; it's baked into the architecture.

Option B we chose — The "hook" approach

Claude Code has a built-in feature: you can tell it to run a script after every shell command finishes. That script receives the command's full output as structured data (stdout, stderr, exit code) and can replace what the agent sees. We inject our script using the --settings flag when launching Claude Code. It runs after the whole command pipeline is done — exactly the right moment.

✅ Decision: Hook wins. We verified live that replacing the output this way actually reduces the number of tokens Claude Code charges you for — not just what appears on screen, but real billed tokens. A test that produced 21,000 tokens of raw output came down to 6,700 tokens with the hook replacing it. 68% reduction in what got charged.

What this means for the cross-agent rail (v2)

The shim approach isn't dead — it's just doing a different job in v2. For the governance and audit ledger, we want to sit at the "action boundary" (when the agent is about to do something, should we allow it?). That's a better fit for the shim. So: hook does compression (v1), shim does governance (v2). Different tools for different jobs.

The one thing that could reverse this decision

If Anthropic changes how Claude Code's hook feature works in a future version, and the replacement stops actually reducing billed tokens, we'd switch to the shim for compression. We have a test that catches this before it silently breaks things.

Design Decision 2 — How is the code structured?

ADR-0002 · 2026-06-16 · Locks all Rust interfaces and module layout

THE QUESTION
What are the exact shapes of the code pieces, and how do they talk to each other?

The three main pieces

The Engine

The brain

Gets the command name, stdout, stderr, and exit code. Decides whether to shorten the output. Returns either a shorter version or "pass this through unchanged." Knows nothing about Claude Code — it's generic.

The Adapter

The glue

Reads the JSON that Claude Code sends via the hook. Translates it for the Engine. Takes the Engine's decision and writes the JSON that Claude Code expects back. Knows about Claude Code's specific format.

The Launcher

The entry point

What runs when you type vajra claude. Sets up the settings file so Claude Code uses our hook. Starts Claude Code. Waits for it to finish. Prints the receipt.

How the Engine decides (the method signature in plain English)

The Engine takes a request: "here is what command ran, and here is its output." It returns a decision: either "here's the shortened version, and I removed N lines" — or simply "don't touch this, pass it through."

The method is called process (not compress), because the engine sometimes decides not to compress anything. Naming it compress would imply it always compresses, which is wrong.

The return is an enum (a named choice, not just true/false), because v2 governance will need a third option — "block this." An enum makes adding that option later free.

Passthrough signal to Claude Code

When we don't want to change anything, we write an empty JSON object {} back to Claude Code. The key for "here's my replacement" is simply absent — Claude Code keeps the original. We do NOT exit with an error code, which might show a warning to the user.

Four things the adapter checks before even calling the Engine

  • Is this actually a shell command? (Not a file-read or other tool type — those we don't touch)
  • Is this an image output? (Can't compress that)
  • Did Claude Code say this command produces no output by design? (Don't touch it)
  • Is VAJRA_RAW=1 set? (User wants no compression — pass through immediately)

How commands are classified (which heuristic to use)

We look at the first word of the command. cargo → use the cargo rules. pytest or python -m pytest → use pytest rules. git → use git rules. Anything else → use the generic "head + tail" fallback.

One important special case: if the command contains &&, |, ;, or similar (it's a pipeline or multi-step command), we skip tool-specific rules and always use the generic fallback. A cargo-specific rule applied to cargo build && echo done output could misread the mixed output.

Folder structure of the codebase

vajractl/ src/ main.rs # Entry point: routes subcommands, catches all errors cli/ hook.rs # "vajractl hook" — the script Claude Code runs after each command launch.rs # "vajra claude" — the launcher engine/ mod.rs # The Engine trait and all shared types (public) default.rs # DefaultEngine: the rules dispatch logic heuristics/ # One file per tool: cargo.rs, git.rs, pytest.rs, npm.rs, generic.rs adapter/ claude_code.rs # ClaudeCodeHookAdapter: reads/writes Claude Code's JSON format meter/ mod.rs # Session cost calculator

Design Decision 3 — How the launcher works and the compression rules

ADR-0003 · 2026-06-16 · Settings injection + all compression heuristics

THE QUESTION
How does vajra claude inject our hook without breaking the user's Claude Code settings? And exactly what are the compression rules?

Part A — How the launcher injects the hook

Claude Code lets you pass a settings file when starting it. Our launcher creates a temporary settings file, puts our hook instruction in it, and passes it to Claude Code via a flag. When Claude Code exits, we delete the temp file. The user's actual Claude Code settings are never touched.

The merge problem: the user might already have their own scripts set up to run after commands. If we just write a fresh settings file with only our hook, we'd overwrite theirs — they'd disappear for the session. So we first read their existing settings from two places (the global settings at ~/.claude/settings.json and any project-level settings at .claude/settings.json), combine them with ours, and write the merged version to the temp file.

Dedup: if vajractl already appears in the user's hooks (maybe they added it manually), we don't add a second copy.

When things go wrong: if we can't read the settings, can't write the temp file, or detect that the settings format has changed in a way we don't understand — we warn with one line to stderr and then just run claude with no hook at all. You get no compression but Claude Code still works normally. We never fail loudly in a way that blocks your work.

Why a file, not a command-line JSON string: JSON strings in shell commands need careful escaping. Files don't. We use a file.

Why spawn + wait, not exec: exec replaces our process with Claude Code's process — our cleanup code never runs and the temp file leaks. With spawn + wait, our process stays alive, Claude Code runs as a child process, and when it exits we clean up properly.

Part B — The two universal thresholds

ThresholdValueWhat it means
Line cap 30 lines If output is shorter than 30 lines, never compress it. Validated by real examples: a 21-line git log passes through; a 46-line directory listing gets compressed. 30 is the right split.
Fail passthrough cap 400 lines If a command failed (non-zero exit) AND produced fewer than 400 lines — pass everything through verbatim. The agent is mid-debug and needs every line. Never compress a small failure.

The order of decisions before any compression happens

Every command goes through these checks in order. The first rule that matches wins; the rest don't run:

  1. Under 30 lines? → Pass through. Don't touch it.
  2. Failed AND under 400 lines? → Pass through verbatim. Agent is debugging.
  3. Compound command (contains &&, |, ;)? → Use generic rules only. Too risky to apply tool-specific rules to mixed output.
  4. Otherwise: → Apply the tool-specific rule for the first word of the command.

How we know if a command succeeded (when there's no exit code)

Claude Code's hook doesn't always pass us the exit code. When it's missing, we look at the last 5 lines of output for unambiguous success signs:

  • "Finished dev profile" or "Finished release profile" → cargo succeeded
  • "test result: ok" → cargo tests all passed
  • "N passed" with no "failed" or "error" nearby → test runner succeeded
  • Anything else → assume it might have failed, and don't compress it

The compression rules per tool

ToolOn success (exit 0)On failure
cargo build Keep only the "Finished dev profile in Xs" line. Drop all 180 "Compiling X v1.2.3" lines. Under 400 lines → full verbatim. Over 400 → keep error lines, drop the compile noise.
cargo test (all pass) Keep the "test result: ok. N passed" summary lines per suite. Drop every individual "test_name … ok" line and all compile output. Keep every FAILED block and its full panic/traceback. Fold passing tests to a count. Add breadcrumb.
git log If large: keep the first 10 commits + "[N more commits]". If under 30 lines: pass through. Rare — pass through.
git diff (with content) Always pass through. The diff IS what the agent is reading. Compressing it removes the information the agent asked for. Always pass through.
git status If large: group by type (modified/untracked/staged), show counts + 3 file samples per group. Pass through.
pytest (all pass) "✓ pytest — N passed in Xs" — one line. Keep FAILED blocks and tracebacks verbatim. Fold passing lines to a count.
npm install Keep the last 5 lines (package count + audit summary). Drop all download/extract progress. Any "npm error" or "ERR!" lines always pass through, regardless of size.
Everything else First 10 lines + last 10 lines, with the breadcrumb header explaining what was cut. Large failures: first 5 + last 15 (bias toward the end, where errors usually appear).

The breadcrumb (every compressed output gets this header)

[vajra: 47 lines folded — VAJRA_RAW=1 before `vajra claude` to see full output] ✓ cargo test — all green in 4.9s lib: 43 passed · bin: 0 · integration: 4 passed · doc: 0

The breadcrumb is the first line, not buried in the middle. The agent immediately knows it's reading compressed output, not a truncated error. If we removed zero lines, we return "pass through" instead — never show an empty breadcrumb.

cargo subcommands that always pass through

cargo fmt, cargo fix, cargo publish — these modify files or publish packages. The agent needs to see exactly what happened. We never compress them. (In practice they often produce no output at all, so the 30-line cap would catch them anyway — but we make it explicit.)

Design Decision 4 — The cost meter and receipt

ADR-0004 · 2026-06-16 · How session cost is measured and shown

THE QUESTION
How does Vajra calculate what a session cost, and how does it tell you?

Where the data comes from

Claude Code writes a log file (JSONL format — one JSON record per line) for every session. It's in ~/.claude/projects/<your-folder-name>/. If Claude Code used subagents during the session, each subagent has its own separate log file in a subfolder. Vajra reads all of them after Claude Code exits. No network needed. No asking Anthropic. The numbers are right there on your disk.

When the receipt appears

After child.wait() — meaning after Claude Code has fully exited and its log files are fully written. We capture a timestamp just before starting Claude Code, then after it exits we find the new log file (any .jsonl file that was created or modified after our start timestamp). That's the session we just ran.

Where the receipt goes — stderr, not stdout

The receipt is information about your session, not data from it. It goes to stderr so that if you pipe Claude Code's output somewhere (vajra claude > output.txt), the receipt appears on your terminal but doesn't end up in the file. Set VAJRA_QUIET=1 to suppress it entirely.

The receipt format

─── vajra · a1b2c3d · 2026-06-16 14:32 ─────────────────────────────
$0.0859 total (opus 163 lines · haiku 8 lines via 2 subagents)
input $0.0128 · output $0.0810 · cache-r $0.0022 · cache-w $0.0139
83 lines folded across 7 tool calls
─────────────────────────────────────────────────────────────────────

Three rows by default: total with model breakdown, token category breakdown, compression summary. Set VAJRA_VERBOSE=1 for a full per-model table.

What the receipt does NOT include — and why

The receipt does not say "you saved $X vs without compression." Computing that would require knowing what the output would have been without compression (the counter-factual), which is approximate and potentially misleading. The receipt tells you what you actually spent. The A/B benchmark (a separate Python script in the bench/ folder) is where the savings comparison happens rigorously — interleaved arms, five runs each, real task completion checks.

How compression line counts reach the receipt

The hook script runs as a subprocess of Claude Code. The launcher runs as the parent. They don't share memory. The solution: before starting Claude Code, the launcher creates a temp file and sets an environment variable pointing to it. The hook script, which Claude Code inherits in its environment, can read that variable and know where to write. After each compression, the hook appends one line to that file: how many lines came in, how many went out, and the first word of the command. When Claude Code exits, the launcher reads and deletes that file. That's where "83 lines folded across 7 tool calls" comes from.

The file records only: line counts and the first word of the command (e.g., cargo). Never the full command string, never any content. This is a hard rule.

Where pricing comes from

The token prices (cost per million tokens for each model) are compiled directly into the binary — there's no config file to manage, no pricing table to download. The same pricing file (bench/pricing.toml) is also used by the benchmark harness. When Anthropic changes prices, we update the file and release a new version.

The schema guard test

If Anthropic changes the format of Claude Code's log file (renames a field, changes a key, removes the two-tier cache breakdown), our cost formula silently gives wrong numbers. To prevent this: we keep a tiny sample log file (bench/fixtures/session_v1.jsonl) with a hand-computed expected cost, and a test that verifies our meter produces that exact number. If the test goes red in CI, we stop and fix the formula before releasing. This test is release-blocking — no green test, no release.

Schema drift fallback If a required field is missing at runtime (Anthropic changed something and the user hasn't updated Vajra yet), we don't crash. We use 0 for that field, flag the receipt with [estimated — update vajractl], and keep going. An imperfect receipt is better than no receipt.

Historical mode

vajractl meter with no arguments lists the five most recent sessions in the current project. vajractl meter <session-id> shows the receipt for that specific session. All the same code — just pointed at a different log file.

All 12 guardrails — things that must be true before v1 ships

These are release-blocking. Not nice-to-haves. If any of these aren't met, we don't ship.

G1
Pin the Claude Code hook contract and verify it still works Detect which version of Claude Code is running. Verify that the hook's "replace output" feature actually does what we expect with a test. If we see an unknown version or the test fails → fall back to passing the original output through. Never guess.
G2
Merge settings, never overwrite When injecting our hook, we always read the user's existing Claude Code settings first and combine them. We never write a settings file that silently removes the user's own hooks or settings. If we can't safely merge → warn and skip injection.
G3
Keep the v2 rail real in CI The compression engine (the brain) is written behind a standard interface so a future governance adapter can plug in. We have a stub test that confirms this interface compiles and works — so the interface doesn't quietly rot before v2 arrives.
G4
Preserve the core invariants VAJRA_RAW=1 completely disables compression. Every compressed output shows a "[N lines folded]" header. Small failing output is never compressed. When in doubt, pass through verbatim and fail open.
G5
Only touch Bash tool output The adapter checks that the tool is actually the Bash tool before calling the compression engine. File reads, web fetches, and other tool types pass through untouched — even if the hook is somehow called for them.
G6
Compress only confirmed successes We only compress when we're sure the command succeeded. Either we got an explicit exit code of 0, or we see an unambiguous success marker in the output (like "Finished dev profile" for cargo). If we're not sure → don't compress.
G7
VAJRA_RAW works at two levels If VAJRA_RAW=1 is set when vajra claude is run → don't inject the hook at all. If it's set any time before the hook runs → the hook exits immediately without compressing. Both layers must work.
G8
Settings merge must cover both global and project settings The settings injector reads from both ~/.claude/settings.json (global) and .claude/settings.json (project-level). If Claude Code's settings format doesn't support multiple hook entries for the same tool, we warn and skip injection rather than overwrite.
G9
Validate the settings format before merging Before reading the user's Claude Code settings and merging them, we check that the settings file is in the format we expect (the hooks list is an array, not some other shape). If it's not → warn and skip, don't try to merge something we don't understand.
G10
Verify the session log folder name matches what Claude Code creates Claude Code names the log folder based on your working directory path. We derive the same name with a simple transformation. A CI test confirms our transformation matches what Claude Code actually creates — so we always look in the right folder for the session log.
G11
The cost formula test is release-blocking We have a sample log file with a hand-computed expected cost. A test verifies our meter produces that exact number. If the test goes red — because Anthropic changed the log format or the formula is wrong — no release goes out until it's fixed.
G12
The compression stats sidecar never records content The file where the hook records compression counts contains only: how many lines came in, how many went out, and the first word of the command (e.g., "cargo"). The full command string, stdout, and stderr are never written to this file. This is non-negotiable.

Four things to verify during coding (can't fully know until we try)

Open question 1

Does --settings add or replace?

If Claude Code's --settings flag adds our settings on top of existing ones, we only need to write our hook to the temp file. If it replaces all settings, we need to copy everything. Test it with a simple experiment on CC v2.1.177.

Open question 2

Does the hook get the exit code?

The verified schema showed stdout, stderr, interrupted flag, isImage, noOutputExpected — but no exit code field. If it's absent, our "confirm success" logic relies on text markers in the output. Test with a real hook call.

Open question 3

Does Claude Code flush the log immediately?

After child.wait() returns, is the log file fully written? Or do we need a brief pause to let the OS flush it? Test with a real session and check timing.

Open question 4

Does {} mean "pass through"?

We're confident that writing an empty JSON object means "don't replace the output" — the ADR-0001 experiment showed that the absent updatedToolOutput key is the clean signal. Confirm it explicitly.

The 5 trust commitments — day one, non-negotiable

1. Zero network calls at runtime When Vajra is doing its job, it never makes a network request. Verifiable by running it offline.
2. No telemetry without explicit opt-in No usage data, no crash reports, nothing — unless you explicitly turn it on.
3. Your code and prompts never leave your machine Session logs are read in-memory. Aggregate numbers only (counts, not content). Nothing is written to disk beyond what Claude Code already writes itself.
4. Fail open by design If anything goes wrong in Vajra — can't parse output, unknown error, CC version we don't recognize — we pass the original unchanged. The agent always gets something. We never block it.
5. Verifiable, signed, reproducible Every release is signed with a public key in the README. SHA-256 checksums on all binaries. cargo install and Homebrew are first-class alternatives — you never have to trust a curl | bash install. Reproducible builds are the stated goal.
The #1 attack surface: the install script The most dangerous thing about Vajra is the day-1 install: curl -fsSL vajra.sh | bash. That's arbitrary code execution. We mitigate it with: pinned/versioned/checksummed installer, signed releases, and always offering cargo/brew/direct-download alternatives. Anyone who doesn't trust the install script has a clean path that doesn't require it.

The traction metric we actually care about

Day-30 retained installs with at least one governance policy actively enforcing. Not stars. Not total installs. Not token-saved counts. The proof that compression on-ramp successfully recruited a governance user — someone who installed for savings and stayed for control.

Naming and release notes

What it's called

Brand: Vajra  ·  Binary: vajractl

The crates.io name "vajra" was already taken, so the Rust package and install command will be vajractl (cargo install vajractl). The brand stays "Vajra."

Confirm ownership of vajra.sh before the public launch. There's a pentest framework with a similar name but the vajractl binary name disambiguates.

Hero artifact for launch

The honest receipt

A 15-second screen recording showing the receipt after a real session. The number is small (~6–8%) and we say so. "Here's why it's not more. Run the benchmark yourself." Honesty is the brand on a skeptical developer audience.

The contributor funnel Each compression heuristic will eventually be a declarative config (TOML rule + golden fixture). "Add support for your favorite build tool" becomes a good-first-issue requiring no Rust knowledge. Every heuristic PR is gated on the fail-open invariant test and the lossless-output invariant test.
